# Contributing to Animation Playground

We welcome contributions from developers of all skill levels! Whether you're fixing bugs, adding features, improving documentation, or suggesting new ideas, your contributions help make Animation Playground better for everyone.

## Getting Started

### Prerequisites

Before contributing, make sure you have:

- **Node.js** (version 18.x or higher)
- **Git** for version control
- **A GitHub account** for submitting pull requests
- **Basic knowledge** of React, TypeScript, and Next.js

### Development Setup

1. **Fork the Repository**

   ```bash
   # Visit https://github.com/abdullahaltun70/animation-playground
   # Click the "Fork" button to create your own copy
   ```

2. **Clone Your Fork**

   ```bash
   git clone https://github.com/YOUR_USERNAME/animation-playground.git
   cd animation-playground
   ```

3. **Add Upstream Remote**

   ```bash
   git remote add upstream https://github.com/abdullahaltun70/animation-playground.git
   ```

4. **Install Dependencies**

   ```bash
   yarn install
   ```

5. **Set Up Environment**

   ```bash
   cp .env.example .env.local
   # Edit .env.local with your development credentials
   ```

6. **Verify Setup**
   ```bash
   yarn dev
   # Visit http://localhost:3000 to ensure everything works
   ```

## Types of Contributions

### ðŸ› Bug Reports

Found a bug? Help us fix it by providing detailed information:

**Before Reporting**

- Check existing issues to avoid duplicates
- Try to reproduce the bug consistently
- Test in different browsers if relevant

**What to Include**

- **Clear title**: Summarize the issue in one line
- **Steps to reproduce**: Detailed steps to recreate the bug
- **Expected behavior**: What should happen
- **Actual behavior**: What actually happens
- **Environment**: Browser, OS, device type
- **Screenshots**: If helpful for understanding the issue

**Example Bug Report**

```
Title: Animation preview not updating when duration changes

Steps to reproduce:
1. Go to playground page
2. Select "Fade" animation type
3. Change duration from 0.5 to 2.0 seconds
4. Observe preview panel

Expected: Preview should show slower animation
Actual: Preview still shows 0.5 second animation

Environment: Chrome 120, macOS Sonoma
```

## Development Workflow

### Creating a Branch

```bash
# Update your local main branch
git checkout main
git pull upstream main

# Create a new branch for your feature
git checkout -b feature/animation-presets
# or
git checkout -b fix/duration-validation
```

**Branch Naming Conventions**

- `feature/description`: New features
- `fix/description`: Bug fixes
- `docs/description`: Documentation updates
- `refactor/description`: Code improvements
- `test/description`: Test additions/improvements

### Making Changes

**Code Quality Standards**

- Follow TypeScript best practices
- Use existing code patterns and conventions
- Write self-documenting code with clear variable names
- Add comments for complex logic

**Testing Requirements**

- Write tests for new features
- Update existing tests when modifying code
- Ensure all tests pass before submitting
- Maintain or improve test coverage

**Running Quality Checks**

```bash
# Run all quality checks
yarn checks

# Individual checks
yarn typecheck    # TypeScript validation
yarn lint         # ESLint checks
yarn prettier     # Code formatting
yarn test         # Test suite
```

### Commit Guidelines

We follow conventional commit format for clear history:

```bash
# Format: type(scope): description
git commit -m "feat(animations): add rotation presets"
git commit -m "fix(preview): resolve duration update bug"
git commit -m "docs(setup): improve installation instructions"
```

**Commit Types**

- `feat`: New features
- `fix`: Bug fixes
- `docs`: Documentation changes
- `style`: Formatting, no code change
- `refactor`: Code restructuring
- `test`: Adding or updating tests
- `chore`: Maintenance tasks

### Submitting Pull Requests

1. **Push Your Changes**

   ```bash
   git push origin feature/your-branch-name
   ```

2. **Create Pull Request**

   - Go to your fork on GitHub
   - Click "Compare & pull request"
   - Fill out the PR template

3. **PR Description Template**

   ```markdown
   ## Description

   Brief description of what this PR does.

   ## Type of Change

   - [ ] Bug fix
   - [ ] New feature
   - [ ] Breaking change
   - [ ] Documentation update

   ## Testing

   - [ ] Unit tests pass
   - [ ] Integration tests pass
   - [ ] Manual testing completed

   ## Screenshots

   (If applicable)

   ## Checklist

   - [ ] Code follows project conventions
   - [ ] Self-review completed
   - [ ] Tests added/updated
   - [ ] Documentation updated
   ```

> ðŸ’¡ **Tip:**
> Small, focused PRs are easier to review and more likely to be merged quickly.
> Consider breaking large changes into multiple PRs.

## Coding Standards

### TypeScript Guidelines

**Type Safety**

```typescript
// âœ… Good: Explicit types
interface AnimationConfig {
  type: 'fade' | 'slide' | 'scale';
  duration: number;
  easing?: string;
}

// âŒ Avoid: Any types
const config: any = { ... };
```

**Naming Conventions**

- `PascalCase`: Components, interfaces, types
- `camelCase`: Variables, functions, methods
- `SCREAMING_SNAKE_CASE`: Constants
- `kebab-case`: File names (except components)

## Issue Labels

Understanding our label system helps you find good first issues:

**Priority Labels**

- `priority: high` - Critical issues affecting many users
- `priority: medium` - Important improvements
- `priority: low` - Nice-to-have features

**Type Labels**

- `bug` - Something isn't working correctly
- `enhancement` - New feature or improvement
- `documentation` - Documentation improvements
- `good first issue` - Good for newcomers

**Status Labels**

- `help wanted` - Extra attention needed
- `in progress` - Someone is working on it
- `needs review` - Ready for code review
